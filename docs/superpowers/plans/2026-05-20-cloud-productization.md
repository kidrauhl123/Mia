# Aimashi Cloud Productization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Turn the current Web/Desktop bridge proof into a real account-based Aimashi Cloud product foundation covering the first seven gaps: unified sync, durable backend storage, basic production security, richer bridge, attachment handling, realtime sync, and desktop cloud sync.

**Architecture:** Replace the JSON-file Cloud store with an embedded SQLite database using Node's `node:sqlite` runtime API. Keep the existing HTTP/WebSocket Cloud server entrypoint, but move persistence and domain mutations into focused modules so Web and Desktop can share account-level data instead of separate local-only state.

**Tech Stack:** Node.js 25, `node:test`, `node:sqlite`, `ws`, Electron IPC, browser WebSocket/EventSource-compatible APIs.

---

## Deliverables And Success Criteria

- Unified account data: conversations, messages, contacts/Fellows, skills, workbench items, files, devices, and bridge runs are scoped by `userId`.
- Durable storage: Cloud data persists in SQLite tables with schema migrations and can bootstrap from the existing JSON data file.
- Security baseline: password hashes remain salted with `scrypt`; sessions are hashed server-side with expiry; file reads verify ownership; auth failures are rate limited per account/IP process window; bridge devices are user-scoped.
- Bridge product behavior: devices report capabilities, selected device runs are recorded, run status is visible, failures/timeouts are persisted, and run requests support text plus attachments.
- Attachments: image uploads and generated images are stored as file rows with ownership, served only to authorized sessions, and rendered without leaking raw server paths.
- Realtime sync: Cloud broadcasts workspace, message, device, and run updates over an authenticated user-scoped WebSocket channel.
- Desktop cloudification: desktop login automatically connects bridge and can pull/push Cloud conversations instead of relying only on separate local state.
