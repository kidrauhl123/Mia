# AION-Style Rust Core Backend Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** On branch `从node技术迁移到rust`, replace Mia's Node-based backend ownership with an AION-style Rust Core. This is an experimental, destructive migration: preserve the desktop UI shell where useful, but do not preserve Node Core compatibility, old local data layouts, or incremental adapter indirection when those slow the backend stack alignment.

**Architecture:** Mia becomes an Electron/renderer UI client plus a Rust Core service. The Rust Core owns settings, providers, bot identity/runtime binding, conversations, task scheduling, MCP, cloud bridge state, process/runtime preparation, persistence, and realtime events. Electron main owns app lifecycle, windows, updater, native dialogs, and launching the Rust Core. Renderer and preload expose UI-facing methods only; they do not assemble model/runtime/provider config, schedule expressions, MCP command records, or agent execution options.

**Tech Stack:** Rust 2024, Tokio, Axum, Tower HTTP middleware, WebSocket realtime events, SQLite via `sqlx`, `serde`/`serde_json`, `tracing`, `anyhow`/`thiserror`, `uuid`, `time`, `cron`, `notify` where needed, Electron/TypeScript-or-CommonJS UI bridge, existing `node:test` only for Electron adapter tests, Rust integration tests for backend behavior.

## Global Constraints

- Work only in `/Users/jung/GitHub/Mia/.worktrees/node-to-rust`; keep `/Users/jung/GitHub/Mia` on `main` untouched.
- This branch is allowed to break old Node Core behavior, local JSON stores, and old IPC contracts if the new Rust Core contract replaces them.
- New backend domain logic must be Rust. New JavaScript may only be Electron shell, renderer UI, thin HTTP/WebSocket adapters, build scripts, or tests around the Electron shell.
- Node/Bun may remain as a managed subprocess runtime only when required by an external tool, matching AION Core's `aionui-runtime` pattern. It must not be the Mia backend owner.
- Renderer and preload must not construct provider secrets, engine-native config, MCP runtime command objects, cron trigger semantics, agent task options, or conversation runtime snapshots.
- Rust Core APIs must use typed request/response DTOs and typed event envelopes; avoid passing arbitrary renderer-owned bags through the system.
- SQLite migrations may create a new destructive schema. A one-time best-effort import path is allowed, but the branch must not be blocked by old data compatibility.
- Every task must include tests or an explicit verification command. Destructive deletions must have a replacement test that proves the new owner path works.

Crate responsibilities:

- `mia-core-db`: SQLite pool, migrations, repositories, transaction helpers.
- `mia-core-realtime`: event bus, WebSocket session registry, event fanout.
- `mia-core-runtime`: subprocess builder, environment sanitization, managed tool location, external agent process spawning.
- `mia-core-system`: health, settings, model/provider registry, permission/runtime policy, desktop environment info.
- `mia-core-memory`: scoped Mia memory search/write/update/forget policy, persistence, and MCP-facing memory tool routes.
- `mia-core-bot`: bot identity, bot runtime binding, assistant snapshots, skill materialization inputs.
- `mia-core-conversation`: conversations, messages, turn orchestration, runtime session state.
- `mia-core-tasks`: one-shot and cron scheduling, task execution, task-to-conversation delivery.
- `mia-core-mcp`: MCP catalog, user server records, OAuth/token storage, connection tests, managed connector supervisor, agent config export.
- `mia-core-cloud`: cloud socket bridge, cloud task/conversation routing, account/session state needed by backend execution.
- `mia-core-app`: `AppServices`, route composition, process entrypoint, graceful shutdown.

## Execution Order

1. Task 1 establishes the branch's destructive architecture decision.
2. Tasks 2 and 3 create the Rust foundation before Electron integration starts.
3. Tasks 4 through 6 cut Electron from Node Core process assumptions and route UI calls to Rust Core.
4. Tasks 7 through 12 move backend domains into Rust. Run these in parallel only when they do not share database migrations or DTOs in the same edit window.
5. Task 13 deletes the old Node backend only after replacement tests pass.
6. Task 14 cleans renderer ownership after Core domain APIs exist.
7. Task 15 is the final gate for the experimental branch.

## Non-Goals

- Preserve the old Node Core as a fallback backend.
- Preserve old local JSON/SQLite data layouts as a compatibility requirement.
- Keep renderer-owned runtime assembly for short-term convenience.
- Build a new frontend framework or replace Electron UI in this branch.
- Remove Node from the Electron app process itself; AIONUI also keeps Electron/TS for the desktop shell.
